# 状态管理的技术选型维度

## 问题

在 React 语境下，状态管理库到底应该按什么维度选，而不是只按“流行”或“语法喜好”来选？

## 简答

先看状态如何被拆分和订阅，再看它与 React 并发渲染、SSR/RSC 的关系，最后再看你要的是手动控制还是更自动的抽象。换句话说，选型维度比“某个库是否火”更重要。

## 综合结论

- 现有材料里最稳定的一组维度来自 `React.md` 和 `State Management.md`：
  - 原子化 vs 非原子化；
  - mutable vs immutable；
  - 手动控制 vs 半自动抽象 vs 自动优化；
  - SSR / RSC 支持；
  - 避免 tearing vs 保留并发渲染收益。
- `State Management.md` 里的 optics 方案代表一种“按状态切片订阅”的方向。它强调：
  - 整体状态仍是 immutable；
  - 通过 optic 把大状态拆成可传递的小引用；
  - 让组件订阅更局部的变化，而不是感知整个 store。
- `zustand` 在现有笔记里对应的是更手动的 external store 路线：
  - 使用简单；
  - 需要明确理解 `Object.is` 和 snapshot 语义；
  - 如果像笔记里那样直接 `Object.assign` 复用对象引用，就可能不触发期待中的更新。
- `React.md` 进一步补上了并发渲染这一维度：外部状态源要么接受 tearing 风险，要么通过 `useSyncExternalStore` 一类同步机制换稳定性，但这又会损失一部分并发收益。
- dated raw 残留里的 2023-04 笔记补了一个很有价值的观察：signals / jotai 之所以吸引人，不只是“语法新”，而是它们试图解决 `useState` 很难共享状态的问题，并把更细粒度的状态单元直接接到视图层。
- 所以从当前材料出发，更稳的选型顺序是：
  1. 先判断你的状态是局部交互态，还是需要跨树共享；
  2. 再判断你更在意局部订阅粒度，还是更在意调试与心智简单；
  3. 最后再看是否需要兼顾 React 并发渲染、SSR / RSC，以及是否愿意接受 tearing 相关折衷。

## 未决问题
