# JavaScript 优化启发

## 问题

JavaScript 性能优化里，哪些启发是值得长期记住的，而不是只在某一篇 benchmark 里偶然有效？

## 简答

真正稳定的优化原则通常不是“某个技巧更快”，而是先少做无用功、保持数据形状稳定、降低间接层和缓存未命中，再用基准测试验证是否真的值得换可读性。

## 综合结论

- 这条记录 汇总的最大价值，在于它把很多零散的 JS 性能建议压缩成几类通用原则：
  - 避免工作；
  - 避免不必要的字符串比较；
  - 避免对象形状频繁变化；
  - 避免额外间接层；
  - 关注缓存命中与数据结构选择。
- 这类原则之所以更值得保留，是因为它们不依赖某个特定框架，而是更接近 JavaScript 引擎和运行时本身的偏好。
- 同时，这条材料也保留了一个重要边界：很多优化会损伤代码可读性，所以 benchmark 不是补充项，而是决定“要不要优化”的前提。
- 因此更稳的顺序不是“先上技巧”，而是：
  1. 先确认瓶颈；
  2. 再选择最小的结构调整；
  3. 最后用基准测试判断优化收益是否真的覆盖了复杂度成本。
