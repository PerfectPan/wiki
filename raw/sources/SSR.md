- 何时用：
	- 更好的 [[SEO]]，当然 SEO 本身由很多因素影响，不是说这里解决了就全解决了。
	- 更早的 FP / FCP
	- 更早的 TTI
	- 通用性强：直出 HTML，低端设备友好，因为 JS 计算量大，无 JS 运行时的环境中，SSR 也能勉强一用。
	- 可选：整个架构是渐进式的，老的 CSR 可以不用或者局部应用。对于旧项目可以作为一个新的性能指标改进，一个新的业务提升点；对于新项目可以在项目支持的情况下直接无缝使用。
- 何时不用：
	- 更关注 TTFB：TTFB 是指客户端和服务端开始交互后，服务端开始向客户端发送数据的时间点，可以用来反映服务器的响应速度。SSR 让服务器承担了数据获取和渲染等工作，对比 CSR，肯定会在更迟的时间点发送数据，因此 TTFB 数据一般不会很好。如果对于这个指标比较敏感，就不要用。
	- 重 UI 交互：就是 JS 准备很久，如果先出了首屏，但是很多操作都无反应就不太好。在一些有复杂交互的网页上，SSR 就显得不那么合适了。或者网页需要 canvas 来呈现内容，那就根本不合适。本来 SSR 将复杂的数据获取搬到了服务器上，客户端可以减少甚至不需要 JS 代码。但如果交互逻辑复杂，就会造成 JS 文件过大，下载时间过长。一旦下载加执行时间超过一定时间（比如 1s） ，用户就会产生困惑，觉得页面出问题了，因为此时的网页是有内容的，但任何操作（除了滑动或者链接点击）都没反应。用户即使多次刷新也无济于事，因为每次走的都是 SSR。
	- 复杂的用户授权：页面访问需要用户授权，某些用户的授权机制依赖浏览器的 Cookie，Local Storage，验证码等，这部分无法放在服务端做，因此不适用。
- 原理：
	- 底层框架提供在服务器上运行框架代码的方式和格式。
	- 上层框架约定好用户代码的写法，然后通过编译器 / 打包器 / 插件等将用户代码打包出一份服务器运行的代码，一份客户端运行的代码。有些框架适配做得很强，甚至可以直接无缝支持原有的 CSR 写法。
- 注意点：
	- 服务器的状态累积：由于代码挪到了服务器上运行，因此同一段代码即可能会多次重复运行，也会在多个用户的上下文下运行，因此如果在代码中出现了 sideeffect，有可能会导致服务器上的状态累积甚至内存泄漏。
	- 不兼容的特性：有一些在浏览器上特有的 API 或者特性，无法直接搬到服务器上运行。有的框架会抹平这些差异，使用框架需要关注。
- SSR 最主要的代码逻辑，就是数据获取和字符串操作（文本渲染）。这部分一般不需要特定 API，轻量的 [[JavaScript Runtime]] 完全可以胜任。借助标准化的 Runtime 的同构，可以做到一次编写，多环境部署。
- 拓展思路：
	- NSR：NSR 是 Native Server Rendering，主要是利用了 Worker（标准化的 Runtime） 的同构性。通过在移动端上嵌入 Worker Runtime，就可以运行与服务端相同的 SSR 项目。移动端可以提前拉取数据做超前渲染，或者在离线情况下仍然提供网站服务。
	- 流式 SSR：通过支持流式渲染，优化 TTFB 指标。
	- CSR 兜底：作为业务兜底保障，框架可以在数据接口超时或者出错时，自动的降级成静态路由，输出普通的 CSR 方案需要的全部文件。
	- SSR 结果缓存：有一些 SSR 输出的在比较长的一段时间都是有效的，那么就可以将这些结果存到存储里。在有效期内的下一次请求就可以立刻响应，省下带宽成本和 CPU 成本。
- 实现思路：
	- 构建时：客户端和服务端产物
	- 运行时：加载服务端入口，数据预取，组件渲染，HTML 拼接（replace 预埋的字符串，还要把数据挂上去让客户端能够同构渲染）
	- 需要考虑的部分：路由管理 / 状态管理 / CSR 降级 / 浏览器 API 兼容 / 自定义 HEAD（Helmet）/ 流式渲染 / SSR 缓存 / 性能监控
- 有必要使用服务端渲染吗：https://www.zhihu.com/question/308792091/answer/573586609
	- 传统渲染直出 HTML 加载 JS 辅助部分动态效果和功能模式：
		- 优点：SEO 友好、首屏渲染快、单个页面来看性能较好（一般 JS 只是辅助，代码量会比较小，服
		  务器性能是可以加钱提升的）
		- 缺点：需要写后端模版、JS 维护困难(跟后端模版业务逻辑绑定）、整站来看性能差（加载需要
		  整个页面重新加载，当然也有 ajax 等技术稍微提升一些体验）
	- 鉴于后端同学并不愿意维护后端模版，前端同学也不是很愿意搞后端那一套，所以就出现了模版前
	  移的运动，属于前后端分离的一部分。这个时期出现了巨量的前端 JS 模版，比如 EJS、Jade、
	  Pug 等，单页应用 SPA 也开始流行起来。后端只负责初始 HTML 页面，给你加载个 JS地址，Js
	  内容和整个页面全部你来搞，顶多再给你输出个全局变量带初始化的数据。
	- 再后面就是React 搞出 JSX 这一套，出来了三大框架，更容易编写富交互的页面，再加上各种
	  Router， 更方便更可维护了一些。然后就进入了现在这个时代。
	- 全 JS 渲染应用模式：
		- 优点：代码全部由一个角色维护、有框架加持方便维护和开发、易于跨终端（毕竟整个应用都是
		  JS)
		- 缺点：SEO 不友好、性能不好 (JS bundle 大，加载完才加载数据然后才运行渲染）
	- 那么 SSR 解决了什么问题？SSR 是在后端拿到了你的 JS 应用代码进行预先计算并解析出来
	  HTML，然后向浏览器吐了 HTML 代码，然后加载框架和你的 JS 应用代码。框架层面支持预读取
	  HTML 结构，来判断是否要重新在客户端渲染还是直接复用。
	- 所以可以得出 SSR 模式：
		- 优点：SEO 友好、性能好（单个页面跟传统一样一次性加载，切换其他页面由 JS 应用代码直接请
		  求渲染，无需整体刷新。还节约了第一次 render 的计算性能放到服务器了）、"模版"复用业务代码
		  Q并没有分散成两块（传统的是后端模版 ＋ 前端JS） 都是一份业务代码
		- 缺点：框架和工程工具缺失、融入或者改造已有项目比较难、涉及跑到 Node 端和客户端，有些业
		  务逻辑要做特殊处理一、压力放在后端一部分，一系列的缓存和运维等问题需要解決、前端要维护
		  一些后端方面的内容(毕竟不是所有后端愿意维护 Node)
- SSR 简单理解就是把你的 React 组件在服务器端渲染了成 HTML，然后塞到一个标签里，到客户端的时候根据你这份 HTML 水合出来，主要是绑定事件，还有还原 React 在内存里的那棵树，所以客户端的代码量其实没有少，一份代码两端都要跑。
	- [[RSC]] 是真正做到把你的部分代码挪到服务端去渲染了，到客户端就没有这部分 JS 大小了，实际上和 SSR 没有特别多的关系，RSC 产出的还是 UI 描述，最终还是要在客户端被渲染成 HTML 的，有 SSR 配合，效果会更好。
