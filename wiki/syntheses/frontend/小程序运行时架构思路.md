# 小程序运行时架构思路

## 问题

小程序这类运行时为什么会不断向“深度定制浏览器内核 + 自己扩展 JS 能力”的方向演进？

## 简答

因为它既想保留 Web 技术栈的开发效率，又不满足于浏览器默认线程模型和能力边界。于是运行时会逐渐把渲染、脚本执行、系统能力和沙箱边界拆开，再在内核层做定制。

## 综合结论

- 这里指向一条很典型的演进路线：
  - 不再满足于单纯使用现成 V8 线程模型；
  - 通过 Chromium 扩展和 worker 化，把运行时执行结构重新切开；
  - 再把 Node 能力和 JSAPI 以受控方式集成进去。
- 这说明小程序运行时的核心矛盾不是“怎么写页面”，而是“如何在 Web 风格开发体验和原生能力控制之间找平衡”。
- 因此更稳的理解是：
  - 运行时不仅是渲染引擎；
  - 它还是能力暴露层、线程调度层和沙箱边界层。

## 未决问题

- 当前材料更像方向性理解，还缺更具体的模块边界、线程模型和宿主 API 设计。
