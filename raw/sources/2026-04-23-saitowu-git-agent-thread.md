## 原始整理

### 1. Git 从来不是面向用户设计的产品

- Scott Chacon 的核心观点之一是：Git 当年根本没想过要做面向用户的产品。
- 它最早是给 Linux 内核团队写的管道命令，本质更接近一组围绕底层版本对象操作长出来的工具。
- 后来虽然从 Perl 脚本逐步改写成 C，但很多命令和交互方式几乎原样保留了下来。
- 结果是，Git 功能很强，但长期没有被当成一个从产品设计出发持续打磨的系统。

### 2. GitHub 与 Git 核心团队一直不是同一种文化

- 帖子提到 GitHub 团队和 Git 核心团队之间曾经存在明显的文化差异。
- GitHub 更像是有 product vision、有 taste 的团队。
- Git 核心团队则更关心工具是否能工作、底层是否稳，并不天然在乎产品交互体验。
- 这种差异也解释了为什么 GitHub 可以成为面向用户的产品，而 Git 本身始终更像一个功能强大的 Frankenstein。

### 3. Git 的原始设计哲学在 agent 时代暴露得更明显

- Git 的设计深受 Unix philosophy 影响：每个命令只做一件事，输出可以 pipe 给下一个命令。
- 但这种设计从一开始就在“机器友好”和“人类可读”之间摇摆。
- 最终结果是，它既不是最佳机器接口，也不是最佳人类界面。
- 对人类开发者来说，这表现为难学、难记、容易踩坑。
- 对 agent 来说，这种问题会被进一步放大，因为 agent 需要不断读取状态、解析文本输出、决定下一步动作。

### 4. Git 对 agent 的几个具体摩擦点

- agent 每执行一个命令，都要重新查询状态。
- 像 `interactive rebase` 这样的流程，对 agent 来说几乎就是噩梦，因为里面充满上下文切换和半结构化交互。
- agent 仍然在用很多老 Unix 工具，例如 `sed`、`grep`，这说明今天的自动化仍依赖旧时代的文本接口。
- Git 没有被设计成一个“稳定的 agent 协作接口”，而更像一个需要经验和上下文才能顺手操作的人类工具。

### 5. 向后兼容让 Git 很难被彻底重做

- Git 几乎永远保持向后兼容。
- 这意味着旧命令、旧行为、旧抽象会长期存在。
- 它保证了生态稳定，但也让“从产品角度彻底重做 Git 交互层”变得极难。
- 难点不只是技术，还有文化：没人想 break anything，也没人真正有权力替整个生态做一次大重写。

### 6. 更现实的方向是：在 Git 之上加一层

- 帖子里提到的答案不是推翻 Git，而是“加一层”。
- 底层保留现有的数据结构、对象模型和传输协议。
- 上层重新设计人类和 agent 的协作体验。
- GitButler 被拿来作为这种思路的例子：它试图补的是产品层，而不是取代 Git 的底座。

### 7. Virtual Branches 是对并行协作抽象的重做

- 一个关键概念是 virtual branches。
- 传统做法里，并行开发往往靠 worktree、多副本或传统分支隔离，再在后面合并。
- GitButler 的思路是，不复制整个工作区，而是在同一代码库上用更高层的抽象隔离不同人的改动。
- 这样做的目标不是消灭版本控制，而是减少传统 merge conflict 和上下文切换成本。

### 8. PR / review 的意义会变化

- 帖子里有个很激进的判断：代码审查已经过时了。
- 更准确地说，过时的不是 review 本身，而是低质量、形式化、噪音很重的 PR 流程。
- 传统 PR 常常堆满：
  - `oops`
  - `fix again`
  - `actually fix`
- 这些 commit 让真正重要的信息被噪音淹没。
- 在 agent 时代，更有价值的 review 可能变成：自动拉取 patch、编译、跑测试、生成摘要，然后由人来验证结果是否可信、方向是否正确、约束是否满足。

### 9. Metadata 会越来越重要

- 帖子里反复强调，未来版本控制可能不再只记录“改了什么”。
- 更重要的是记录：
  - chat transcript
  - tool call logs
  - spec
  - 决策背景
  - ownership / coordination 信息
- 换句话说，代码只是协作的一部分；过程、约束和理由会越来越重要。
- 但这也带来新的难题：并不是所有 metadata 都值得保存，过量记录会导致信息爆炸和 UI 不可用。

### 10. agent 时代更稀缺的是写作、沟通和定义问题的能力

- 帖子里最重要的判断之一是：未来最强的工程师，是那些能沟通、能写作、能清晰描述的人。
- 当 AI 让“how”变便宜后，真正稀缺的变成：
  - why
  - what
  - spec
  - constraints
- 能不能把目标、边界、验收标准讲清楚，会直接决定 agent 协作的质量。
- 这意味着未来工程师的核心竞争力，会从“亲手写每一行实现细节”部分转移到“能不能把复杂系统的协作问题定义清楚”。
