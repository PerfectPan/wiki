# 程序语言理论与实现公开课

- 每种语言的抽象都有一定的 cost
- codegen 阶段：[[栈]]的访问速度是大于[[堆]]的访问速度，[[寄存器]]的访问速度是大于栈的访问速度，会基于此做一些处理
- 很多语言会编译到 [[LLVM]] 后端，然后都交由 LLVM 去做处理
- [[IR]] 会有一步线性化，因为 [[CPU]] 是接受线性的指令集（优化表述？）
- 局部变量的语义与编译过程
	- 变量名是静态的，可以考虑抹掉
	- expr -> nameless expr（递归使用了宿主语言的栈）-> 栈式虚拟机指令
- 栈式虚拟机：接受一堆平行的指令，丢进去执行，一个 while loop
- 编译器把 局部变量 / 参数 / 临时变量 / stack frame 四个栈都合并为一个栈
- expr -> Nameless.expr -> Indexed.expr（主要是中间会有临时的操作数，所以需要明确它在栈上的局部变量名的位置） -> linearize -> StackMachine
- 局部变量名 | 临时存储的操作数(Add, Mul 啥的)
- 实际实现的 stack 是一个 immutable 的数据结构，因此不用考虑数据还原的问题
- 现在内存大了，可以支持 multiple IR，以前是追求一步到位
	- 引入函数
		- 作用域包含动态作用域（运行时决定）和静态作用域（根据代码位置确定），现代语言更倾向静态作用域（lexical scope）
		- 比原来的 language 多保存上下文
		- 闭包目前会捕获多余的变量
- 语言变大后就不好形式化研究，需要简化语言
	- $$Let(x, e1, e2) == App((Fn(x, e2), e1)$$
		- 在理论上等价，但是前者效率会更高
	- Currying
		- $$Fn(x_1,x_2,e)== Fn(x_1,Fn(x_2,e))$$
		- $$App(e,e_1,e_2)==App(App(e,e_1),e_2)$$
- ```rescript
  // reinvent the Lambda calculus
  type rec lambda = 
    | Var(string)
    | Fn(string, lambda)
    | App(lambda, lambda)
  ```
- [[Lambda 演算]]足够小，可以来做形式化的证明
- TODO Programming languages and lambda calculi #Reading
- [[draws/2022-11-24-13-20-57.excalidraw]]
- 类似 $\beta$ reduction 的替换方式常用于编译器优化，本身效率不高
- 用环境存储上下文变量用于高效的解释器实现
- 德布朗指数表示变量
	- 名字不重要
	- $\lambda x.\lambda y.x(y(x))$ 可以被表示为 $\lambda .\lambda .1(0(1))$
	- 数字 i 代表被这个变量被第 i 个 $\lambda$ 表达式绑定
	- 不用考虑重名问题但是很难处理 / 实现
	- 对优化不友好
	- 执行替换的时候进一层 $\lambda$ 表达式被替换数的值就要加一，同时替换的表达式里的自由变量也要加一
	- shift_aux(i, d, t)
		- $(j)=j,\rm{if}\ j < d$
		- $(j)=i+j,\rm{if}\ j >= d$
		- $(\lambda .t)=\lambda .(t)$
		- $(t_1t_2)=(t_1)(t_2)$
- calling convention
	- 在 caller 和 callee 之间的协议：栈和寄存器如何组织，当函数 call 发生的时候
- 指令集放到虚拟机上跑，汇编语言编译到机器码，包含一个 opcode（操作码） 和若干操作数
- 栈平衡原则即一段表达式执行完后只剩下一个结果的数
