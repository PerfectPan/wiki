# 服务端与客户端视图一致性

## 问题

为什么在 SSR / hydration 场景里，服务端和客户端渲染出不同视图会很危险？如果确实必须不同步，应该怎么做？

## 简答

因为 hydration 默认假设服务端 HTML 和客户端第一次渲染结果一致。只要这条假设被破坏，React 就要在接管阶段额外修补，甚至直接重建部分树。真正无法一致时，更稳的做法是把差异延后到 effect 或客户端阶段，而不是在首帧就让两边分叉。

## 综合结论

- 重点不是某个 API，而是一条约束：避免让服务端和客户端在首轮渲染时直接走出两套不同视图。
- 这条约束的重要性在 SSR / hydration 场景里尤其高，因为客户端接管时默认认为已有 HTML 是可信基线。
- 如果一开始两边就不一致，问题不只是“看起来闪一下”，还可能导致：
  - hydration 警告；
  - 非预期重建；
  - 更难推理的交互边界。
- 因此更稳的处理方式是：
  - 尽量让首轮输出保持一致；
  - 真正依赖浏览器环境的差异，推迟到 `useEffect` 或客户端专有阶段；
  - 把“不同步”变成显式、受控的后续更新，而不是首帧就分叉。
